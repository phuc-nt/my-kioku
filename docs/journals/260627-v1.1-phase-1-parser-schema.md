# v1.1 Phase 1 — Parser & Schema (relations + tags)

**Date:** 2026-06-27 · **Commit:** 0b50d58 · **Branch:** feat/v1.1-emotional-relations

## What shipped

The parse + schema foundation for emotional relations. Additive on v1.

- `inline-field-parser.ts` (NEW): strict `parseRelationLine` (`joy:: [[a]], [[b]]`) + `parseTagsLine` (`tags:: a, b`).
- `entry-parser.ts`: `extractMood` → `extractLeadingFields` loop — consumes a contiguous run of mood/relation/tags lines, stops at the first prose line.
- `db.ts`: `relations(entry_id, rel_type, target)` + `tags(entry_id, tag)` tables + indexes; SCHEMA_VERSION 2→3.

## Key decisions / learnings

- **Verbatim invariant is the whole safety story** — code review threw ~40 adversarial inputs (trailing punctuation, glued suffixes, unbalanced/nested brackets, blank-line-stops-zone, field-after-body, `tags:: [[x]]`, invalid-mood-stops-zone) and **every one stayed verbatim**. The strict residue rule (strip `[[...]]` + separators → must be empty) is what makes `with:: my friend Hùng` (prose) stay in the body while `with:: [[Hùng]]` becomes a relation.
- **M1 (fixed)**: tags were concatenated without de-dup while relations de-duped → asymmetry that would inflate Phase 2 rows + reflect tag counts. Now both de-dup + preserve order.
- **L1 (fixed proactively)**: pre-existing CRLF leak (body kept trailing `\r`). Normalized `\r\n` in `parseEntries` — also de-risks Phase 5 (Telegram files could be CRLF).
- **L2 (intentional)**: any non-reserved verb is a valid relation (`time::`, `text::`) — free-form by design; only `mood`/`tags` reserved.
- Files stay <200 LOC (entry-parser 163, inline-field-parser 61, db 141) — matchers split into their own module, reused by entry-parser (DRY; remember/import will reuse in later phases).
