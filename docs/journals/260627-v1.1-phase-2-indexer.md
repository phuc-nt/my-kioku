# v1.1 Phase 2 — Indexer (relations + tags rows)

**Date:** 2026-06-27 · **Commit:** 0b0b960 · **Branch:** feat/v1.1-emotional-relations

## What shipped

Persist parsed relations/tags into their tables — same transaction as entries/fts/links.

- `indexJournal`: one `relations(entry_id, rel_type, target)` row per (verb, target), one `tags(entry_id, tag)` per tag.
- `removeFile`: deletes relations/tags by file-keyed entry-id subquery, **before** the entries delete.
- `fullReindex`: truncate list + `ReindexStats` now include relations/tags counts.

## Key decisions / learnings

- **Delete ordering is load-bearing**: relations/tags DELETE must precede `DELETE FROM entries` — else the `entry_id IN (SELECT id ... WHERE file = ?)` subquery returns nothing → orphan rows. A code comment marks this where a future editor would break it.
- **Code review verified all high-risk axes safe** (with executed repros): orphan-on-reindex (stale rows cleared), dup-date partial-txn (failed entry's relations/tags don't leak — `insEntry` throws first), counts accurate, CRLF-normalize a byte no-op on LF input, atomic via `db.transaction`.
- **Test-coverage gaps closed (M#1, M#2)**: the original "dup-date safety" test used different dates + one `fullReindex` → never exercised Invariant #4. Rewrote it to edit one file and re-index via the **single-file `indexFile` path** (what lazy-sync uses) — asserts stale rows removed, other file's rows survive, no orphans. This is the only path where a delete-ordering regression could hide; Phase 3 builds incremental sync on it.
- indexer.ts now **199 LOC** (at the 200 limit) — any further journal-row logic must extract to `journal-rows.ts`.

## Verification

180 tests, tsc clean. Smoke: reindex reports `relations: 2, tags: 2`. Rebuild-identical confirmed v1 vaults unaffected.
